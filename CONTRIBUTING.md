# Contributing Guide

We are very happy that you would like to contribute. In this guide you will find instructions on how to setup the repo locally.

## Setup

### Install

#### 1. Install Bun

To be able to run all tests in the "./packages/global-registrator" package, you need to install [Bun](https://bun.sh/).

<details>
  <summary>Linux & MacOS</summary>
    ```bash
    curl -fsSL https://bun.sh/install | bash
    ```
</details>
<details>
  <summary>Windows</summary>
    ```bash
    powershell -c "irm bun.sh/install.ps1 | iex"
    ```
</details>

#### 2. Install dependencies

```bash
npm install
```

### Compile

```bash
npm run compile
```

### Watch

```bash
npm run watch
```

### Test

**Run tests**

```bash
npm test
```

**Watch tests**

```bash
npm run test:watch
```

### Debug

1. Go to the package you wish to test and debug in the terminal (e.g. `cd ./packages/happy-dom`)
2. Write "debugger;" at the place you want to place a breakpoint in the code
3. Run `npm run test:debug` in the terminal
4. Open Chrome
5. Open developer tools
6. A green ball should appear to the left of the menu bar in developer tools
7. Click on the green ball
8. Click continue to jump to your breakpoint

# Pull Request Guidelines

Checkout a topic branch from a base branch, e.g. main, and merge back against that branch.

## Title

The title of the pull request should be in the format of "type: [#issue] description". The type can be one of the following:

- feat: A new feature
- fix: A bug fix
- chore: A change that does not fix a bug or add a feature
- BREAKING CHANGE: A change that breaks backward compatibility

The issue is optional and can be omitted if the pull request does not relate to an issue.

## Commit Convention

There is no strict commit convention, but it's recommended to follow the same format as the pull request title for commit messages. This will make it easier to understand the purpose of each commit and will help with generating changelogs.

It's OK to have multiple commits in a pull request. The commits will be squashed into one commit when the pull request is merged.

## Review

All pull requests must be approved by @capricorn86 who is the founder and main maintainer of the project. Other maintainers and contributors can also approve pull requests, but their approval is not required for the pull request to be merged. Hopefully in the future we will have more maintainers who can approve pull requests.

The review process may take some time, so please be patient. If you have any questions or need help with your pull request, please ask for help in the pull request or in the issue.

# Tests

Make sure to add tests for your changes. Run your test in a real browser to make sure that the test tests what a real browser would do (e.g. by running the code in the browser console).

Pull requests will not be merged if they do not have tests that cover the changes. If you are not sure how to write tests for your changes, please ask for help in the pull request or in the issue and a maintainer will help you.

## If it's a feature

Add tests that cover the new feature and all edge cases you can think of. You only need to test the public API. Internal functionality should be tested through the public API.

One way to do this is to read the the added code line by line and whenever you see a new function or a new code path, ask yourself "how can I test this?" and then write a test for it.

## If it's a bug fix

Add a test that reproduces the bug and then fix the bug. The test should fail before the bug is fixed and should pass after the bug is fixed.

# AI Contributions

We welcome AI as a personal assistant when contributing. However, we strongly believe that a real person must be behind every issue and pull request. The code itself is not the most important part. If it where, everything for Happy DOM would have been automated already.

You are responsible for the content of your contribution, even if it was generated by AI. Make sure to review and test the generated code to ensure it meets the project's standards.

All issues and pull requests must be opened by a real person using the official templates. If AI assisted in creating a pull request, please disclose the tool used (e.g. Claude, Codex, Copilot).

Pull requests or issues entirely generated by AI with no human involvement (e.g. by an automated agent) will be labeled "maybe automated" by the maintainers and will be closed after 3 days, unless a real person responds. The response is expected to be genuine, meaning it should not be written by an LLM. The maintainers reserve the right to decide, based on their experience, whether a response was AI-generated.

These measures help reduce maintenance burden and keep work efficient.

# Coding Guidelines

Happy DOM is written in TypeScript using an object-oriented programming style. We use classes and inheritance to structure the code. We also use interfaces to define the public API of the classes in some cases. Utility functions are also defined in classes, usually as static methods. The DOM functionality contains a lot of complex functionality, so we organize it into classes to make it easier to understand and maintain (e.g. `File.getName()` gives context compared to `getName()`).

The code is documented with JSDoc comments. The comments should be clear and concise, and should explain the purpose of the code and how to use it. The comments should also include examples if necessary. The JSDoc comments are part of the public API and should be written in a way that is easy to understand for users of the library.

When making a contribution, try to follow the existing coding style and conventions as much as possible. This will make it easier for the maintainers to review your code and will increase the chances of your pull request being merged.

# Dependencies

Happy DOM aims at being lightweight and fast, so we try to minimize the number of dependencies. We only use dependencies that are necessary for the functionality of the library. If you want to add a dependency, please make sure to justify why it is necessary and how it will benefit the library.

Each dependency adds maintenance burden and can potentially introduce security vulnerabilities, so we are careful with adding dependencies.

It's many times better to write a few lines of code yourself than to add a dependency for something that can be easily implemented. If you are not sure whether a dependency is necessary, please ask for help in the pull request or in the issue.

# Release Process

Releases are triggered manually by the maintainers. The release process is as follows:

1. Merge all pull requests that are ready to be released into the main branch.
2. Trigger the release workflow, which will generate a new version based on the commit messages and publish it to NPM.
3. A GitHub release will be created with the changelog and the new version.
4. The new version will be available on NPM.
